# Ship Check

Audit a project directory before it leaves your machine. Catch anything embarrassing, bloating, or broken before the recipient opens it.

A delivery should contain exactly what's needed to understand, build, and run the project — nothing more. Every extra megabyte of node_modules signals carelessness. Every leaked .env is a security incident.

## Process

### 0. Read Gotchas
Read `gotchas.md` in this skill's directory (if it exists) before proceeding.
Known failure modes inform execution — avoid repeating past mistakes.

### Workflow

#### 1. Identify the Delivery Directory

The user may want to ship the entire repo, a subdirectory, or a specific path. If unclear, ask. Everything below targets this directory.

#### 2. Detect the Stack

Scan for project manifests to understand what's in play:

| Marker | Stack | Common Artifacts to Exclude |
|--------|-------|-----------------------------|
| `package.json` | Node.js | `node_modules/`, `build/`, `dist/`, `.next/` |
| `requirements.txt`, `pyproject.toml` | Python | `venv/`, `.venv/`, `__pycache__/`, `*.pyc`, `*.egg-info` |
| `Cargo.toml` | Rust | `target/` |
| `go.mod` | Go | (`vendor/` sometimes intentional) |
| `Gemfile` | Ruby | `vendor/bundle/` |
| `Dockerfile` | Docker | delivery may be image not source — ask |

A project may have multiple stacks. Detect all of them.

#### 3. Audit

Use the **Agent** tool to dispatch up to 3 Explore agents in parallel, distributing these categories across them for speed.

##### A. File System Hygiene

**Dev artifacts that must not ship** (recipients install their own):
- `node_modules/`, `venv/`, `.venv/`, `env/`, `__pycache__/`, `target/`
- `.DS_Store`, `Thumbs.db`
- `.idea/`, `.vscode/` (unless project settings are intentional)
- `*.log` files
- `.claude/`, `claude_dev/`, `project_files/`

**Build outputs** (`build/`, `dist/`, `out/`) — usually regenerated by the recipient. Flag them; the user decides whether to include.

**Large files** (>10MB) not explicitly needed — model weights not in LFS, database dumps, videos, vendored binaries. Report their sizes.

**Git LFS** — verify tracked files are actual content, not pointer stubs. This matters because `zip` ships the working copy (real files), but a `git archive` or shallow clone may only have stubs.

##### B. Security & Secrets

**Before running security checks:** Read `$BRAIN_ROOT/_Docs/<slug>/Reports/audit-context.json` and the most recent `$BRAIN_ROOT/_Docs/<slug>/Reports/security-*.md` (check today's date first, then fall back to most recent).

**If a security audit report exists from the last 7 days:**
- **Skip** secrets-in-code scanning (SEC- findings already cover this thoroughly)
- **Skip** security config checks (CFG- findings already cover this)
- **Keep** the file-based checks below — these are delivery-specific (should this FILE ship?)
- **Report** the security audit state: "Security audit from YYYY-MM-DD: N findings FIXED, M still OPEN. Run `/security-audit --resume` to address open findings before shipping."
- If there are OPEN CRITICAL or HIGH findings, escalate to BLOCK: "N critical/high security findings are still open — address before shipping."

**If no recent security audit exists** (older than 7 days or never run):
- Run the full Section B below as a safety net
- Add a WARN: "No recent security audit found. Run `/security-audit` for a thorough security review."

**The following file-based and code-pattern checks run in all cases (delivery-specific). Secrets-in-code scanning and security config checks below only run in the fallback path (no recent security audit).**

**Files that must not ship:**
- `.env` (real config — `.env.example` / `.env.template` are fine)
- `credentials.json`, `serviceAccountKey.json`, `*.pem`, `*.key`
- Any file with "secret", "password", "token", or "credential" in the name

**Secrets in code** — use the **Grep** tool to search for patterns:
- `sk-`, `ghp_`, `AKIA`, `Bearer ` (with trailing space)
- `password\s*=`, `secret\s*=`, `token\s*=` (assignment, not usage)
- Hardcoded API keys or database connection strings

**Security config concerns:**
- CORS set to `*` in production config
- Auth disabled or hardcoded bypass
- Debug mode enabled in production settings

##### C. Code Cleanliness

- `console.log` / `console.debug` in JS/TS (excluding files named `logger` or log utility modules)
- `print()` used for debugging in Python (excluding `if __name__` blocks and CLI tools)
- `debugger` statements
- `TODO`, `FIXME`, `HACK` comments
- Hardcoded `localhost` in non-config source files (config files and .env templates are fine)

##### D. Documentation & Config

**Must-haves:**
- `README.md` — exists, is substantive (not a placeholder), has setup instructions
- Environment template — `.env.example` or equivalent with all required vars documented
- `.gitignore` — exists and covers the detected stack's artifacts

**Nice-to-haves:** License file, architecture overview, troubleshooting section.

##### E. Git State

Use the **Bash** tool to run git status checks:

- Uncommitted modifications or untracked files that belong in the repo
- New files that should be tracked (fonts, configs, documentation)
- Whether the current branch is the intended release branch

#### 4. Report

Compile findings into a readiness report with three severity levels:

- **BLOCK** — Must fix before delivery. Secrets exposed, missing README, uncommitted core changes.
- **WARN** — Should fix but delivery is functional. TODOs in code, .DS_Store files, suboptimal docs.
- **OK** — Checked and clean.

Use this structure:

```
## Ship Check: [directory name]

### Verdict
[Ready / Needs work / Not ready]

| Metric | Value |
|--------|-------|
| Current dir size | X MB |
| Estimated clean zip | ~X MB |
| Bloat | X MB in N items |

### BLOCK (N issues)
- [ ] Issue — path or details
  → Fix: `command or instruction`

### WARN (N issues)
- [ ] Issue — path or details
  → Fix: `command or instruction`

### Passed
- [x] Category — what was checked and passed
```

#### 5. Generate Commands

After the report, provide two code blocks the user can review and run:

**Cleanup block** — commands to fix all flagged issues:
```bash
# Delete OS artifacts
find . -name '.DS_Store' -delete
# Clean temp/test data
rm -f path/to/uploads/*.jpeg ...
```

**Zip block** — a zip command with exclusions matching the detected stack:
```bash
zip -r project-name.zip project-dir/ \
  -x "project-dir/node_modules/*" \
  -x "project-dir/.env" \
  -x "*/.DS_Store" \
  -x "*/__pycache__/*" \
  ...
```

Include the expected clean zip size estimate.

#### 6. Commit Reminder

If there are uncommitted changes that should ship, list them with file paths and suggest a commit message. Remind the user to commit before zipping — a zip snapshots the working directory, so uncommitted changes are included, but it's better practice to have a clean git state that matches what's delivered.

## Rules

### Edge Cases

- **Monorepo subdirectory delivery**: Audit only the delivery directory, but check for symlinks or relative imports that reference files outside it — these would break when extracted standalone.
- **Docker delivery**: If there's a Dockerfile and the user intends to ship an image (not source), the audit shifts — check Dockerfile best practices, multi-stage builds, .dockerignore instead of source-level cleanup.
- **Intentionally large files**: Model weights, datasets, or media that are part of the product — flag them with their size but don't auto-exclude. Ask the user whether they should ship.
- **No git repo**: Skip the git state check entirely but warn that there's no version control tracking what was delivered.
- **Pre-built delivery**: If the user intends to ship compiled output (not source), adjust expectations — build artifacts become the product, not bloat.
